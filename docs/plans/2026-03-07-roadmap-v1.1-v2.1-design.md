# Roadmap v1.1 - v2.1 Design

## v1.1 — Rector + Extension-Scan + Visualisierung

### Rector-Rule-Skeleton-Generator

**Neue Komponenten:**
- `src/Generator/RectorConfigGenerator.php` — Generiert `rector.php` Config fuer einfache Faelle
- `src/Generator/RectorSkeletonGenerator.php` — Generiert Rule-PHP-Dateien fuer komplexe Faelle
- `src/Dto/RectorRule.php` — VO mit Typ (config vs. skeleton), Quell-/Ziel-Referenz, Before/After Code

**Mapping-Logik:**

| RST-Pattern | Rector-Ausgabe |
|---|---|
| Klasse umbenannt (Alt in Description, Neu in Migration) | Config: `RenameClassRector` |
| Methode umbenannt | Config: `RenameMethodRector` |
| Statische Methode umbenannt | Config: `RenameStaticMethodRector` |
| Konstante umbenannt | Config: `RenameClassConstFetchRector` |
| Methode entfernt ohne Ersatz | Skeleton mit TODO |
| TCA-Aenderung | Skeleton mit TODO |
| Hook -> Event | Skeleton mit TODO |

**Erkennung Alt->Neu:** Der RstParser wird erweitert um aus dem Migration-Abschnitt "Replace X with Y" Paare zu extrahieren. Regex auf Patterns wie:
- "Replace :php:`OldClass` with :php:`NewClass`"
- "Use :php:`NewMethod()` instead"
- "has been renamed to :php:`NewName`"

**UI:** Neuer Tab auf der Matcher-Generate-Seite — neben "Matcher-Config" auch "Rector-Config" und "Rector-Skeleton" mit Preview + Download.

### Extension-Scanner

**Neue Komponenten:**
- `src/Scanner/ExtensionScanner.php` — Scannt PHP-Dateien einer Extension gegen Matcher
- `src/Scanner/ScanResult.php` — VO mit Findings (Datei, Zeile, Matcher, Severity)
- `src/Scanner/SourceProvider.php` — Interface fuer Pfad/Git-Quellen
- `src/Scanner/LocalPathProvider.php` — Liest von lokalem Pfad
- `src/Scanner/GitRepositoryProvider.php` — Klont Repo temporaer, raeumt auf
- `src/Controller/ScanController.php` — Formular + Ergebnis-Ansicht

**Scan-Ansatz:** Nutzt die bestehenden TYPO3-Matcher-Klassen aus `typo3/cms-install` direkt — die sind als `NodeVisitor` fuer `nikic/php-parser` implementiert. Wir instanziieren sie mit unseren Matcher-Configs + den generierten Configs und lassen sie ueber die Extension-Dateien laufen.

**UI:**
- Eingabe-Formular: Pfad-Feld oder Git-URL-Feld
- Ergebnis-Tabelle: Datei, Zeile, betroffene API, Typ (Deprecation/Breaking), Link zum RST-Detail
- Export als JSON/CSV

### Before/After Code-Vergleich

**Parser-Erweiterung:**
- `RstParser` erweitern um `.. code-block:: php` Bloecke im Migration-Abschnitt zu extrahieren
- `src/Dto/CodeSample.php` — VO mit `before: ?string`, `after: ?string`, `language: string`
- `RstDocument` bekommt ein neues Feld `codeSamples: CodeSample[]`

**UI:** Side-by-Side Darstellung auf der Deprecation-Detail-Seite. CSS Grid mit `grid-template-columns: 1fr 1fr`, auf schmalen Screens `1fr`. Syntax-Highlighting via CSS (keine JS-Library noetig fuer v1.1, einfaches `<pre><code>`).

### Coverage-Report

**Neue Komponenten:**
- `src/Analyzer/CoverageBreakdownAnalyzer.php` — Berechnet Coverage nach Dimension
- `src/Dto/CoverageBreakdown.php` — VO mit Maps: `byVersion`, `byType`, `byExtension`, `byScanStatus`
- `src/Controller/CoverageController.php` — Report-Seite

**UI:** Dashboard-Seite mit:
- Balkendiagramm Coverage pro Version (CSS-only, keine Chart-Library)
- Tabelle: Version x Typ Kreuzmatrix
- Aufschluesselung nach `ext:*` Tags
- Filter: Version, Typ, Extension kombinierbar

## v1.2 — Intelligentere Analyse

### Argument-Erkennung

**Parser-Erweiterung:**
- Aus `.. code-block:: php` Bloecken im RST Methodensignaturen extrahieren
- Regex auf Funktionsdefinitionen: `function methodName(Type $param1, ?Type $param2 = null)`
- Daraus `numberOfMandatoryArguments` und `maximumNumberOfArguments` berechnen
- Alternative: Reflection auf die TYPO3-Klassen via `symfony/property-info` — wenn die Methode in der installierten Version noch existiert, Signatur direkt auslesen

**Integration:** `MatcherConfigGenerator` nutzt die erkannten Argumente statt 0/0 Defaults.

### Migrations-Mapping

**Neue Komponenten:**
- `src/Analyzer/MigrationMappingExtractor.php` — Extrahiert Alt->Neu Paare aus RST
- `src/Dto/MigrationMapping.php` — VO: `source: CodeReference`, `target: CodeReference`, `confidence: float`

**Patterns** (aus dem Migration-Abschnitt):
- "Replace :php:`X` with :php:`Y`" -> confidence 1.0
- "Use :php:`Y` instead" -> confidence 0.9 (Kontext-abhaengig)
- "has been renamed to :php:`Y`" -> confidence 1.0
- "Migrate to :php:`Y`" -> confidence 0.8

**Integration:** Rector-Generator nutzt Mappings fuer automatische Config-Generierung. UI zeigt Mappings auf der Detail-Seite an.

### Komplexitaets-Scoring

**Neue Komponenten:**
- `src/Analyzer/ComplexityScorer.php` — Bewertet jedes RST-Dokument
- `src/Dto/ComplexityScore.php` — VO: `score: int (1-5)`, `reason: string`, `automatable: bool`

**Scoring-Kriterien:**

| Kriterium | Score |
|---|---|
| Klasse/Methode renamed, 1:1 Mapping | 1 (trivial) |
| Methode entfernt mit klarem Ersatz | 2 (einfach) |
| Signatur-Aenderung (Argumente) | 3 (mittel) |
| Hook -> Event Migration | 4 (komplex) |
| TCA-Struktur-Umbau | 4 (komplex) |
| Architektur-Aenderung ohne klaren Ersatz | 5 (manuell) |

**Integration:** Auf allen Listen als farbige Badges (1=gruen, 5=rot). Sortierbar. Filter "nur automatisierbare".

## v1.3 — Extension-Scanner als Service

### ZIP-Upload

- Erweiterung von `SourceProvider` Interface um `ZipUploadProvider`
- Symfony Form mit File-Upload, Validierung (max 50MB, nur .zip)
- Temporaeres Entpacken in `var/tmp/`, automatisches Cleanup nach Scan

### Findings-Report

- Erweiterung von `ScanResult` um detailliertere Informationen
- Code-Kontext: 3 Zeilen vor/nach dem Finding anzeigen
- Gruppierung nach Datei, nach Severity, nach Deprecation-Version
- Zusammenfassung: "X Findings in Y Dateien, davon Z automatisch behebbar"

### Export

- JSON-Export: Strukturierte Daten fuer Tool-Integration
- CSV-Export: Fuer Excel/Projektmanagement
- Markdown-Export: Fuer Jira/GitHub-Issues

## v2.0 — Multi-Version + Rector-Integration

### Multi-Version Support

**Architektur-Aenderung:**
- `DocumentService` bekommt konfigurierbaren Versions-Bereich
- TYPO3 als Composer-Dependency mit Versionsmatrix:
  - v2.0-branch mit `typo3/cms-core:"^13.4"` (enthaelt Changelogs 7.x-13.x)
  - Alternativ: RST-Dateien als separates Datenpaket auslagern (Git-Submodule oder eigenes Composer-Paket)
- UI: Version-Range-Selector im Dashboard ("Zeige Migration 11->12" / "12->13")

### Lauffaehige Rector-Rules

- Fuer einfache Patterns (Rename, Remove Argument) komplette Rule-Klassen generieren
- Nutzt `nikic/php-parser` Node-Typen direkt
- Test-Skeleton mit Before/After aus RST als PHPUnit-Fixture
- Generierte Rules in einem temporaeren Verzeichnis, per Download oder als Composer-Paket

### Rector-Config-Export

- Komplette `rector.php` generieren die alle noetigen Rules fuer eine Extension enthaelt
- Basierend auf Scan-Ergebnissen: nur Rules die tatsaechlich gebraucht werden
- Integration mit `ssch/typo3-rector`: pruefen welche Rules dort schon existieren, nur fehlende generieren

### Diff gegen ssch/typo3-rector

- `ssch/typo3-rector` als optionale Composer-Dependency
- Abgleich: welche Deprecations sind durch typo3-rector Rules abgedeckt?
- Report: "typo3-rector deckt X% ab, Y Rules fehlen, Z koennten generiert werden"

## v2.1 — CI/CD Integration

### CLI-Modus

- Symfony Console Commands:
  - `bin/console scan:extension /path/to/ext` — Extension scannen
  - `bin/console scan:extension --git=https://github.com/...` — Git-Repo scannen
  - `bin/console report:coverage` — Coverage-Report als Text/JSON
  - `bin/console generate:matcher` — Fehlende Matcher generieren
  - `bin/console generate:rector` — Rector-Config generieren
- Exit-Codes: 0=clean, 1=findings, 2=error
- JSON-Output-Modus (`--format=json`) fuer Tool-Integration

### GitHub Action

- `magicsunday/typo3-migration-check-action`
- Konfiguration: TYPO3-Version, Schwellwerte, Extension-Pfad
- PR-Kommentar mit Findings-Zusammenfassung
- Status-Check: fail bei Breaking Changes

### Composer Plugin

- `magicsunday/typo3-migration-check-plugin`
- Hook in `post-install-cmd` / `post-update-cmd`
- Prueft ob Extension-Code gegen die installierte TYPO3-Version kompatibel ist
- Warnung bei Deprecations, Fehler bei Breaking Changes
